# Purpose

This document provides an engineering-oriented overview of the IEC 62443 series from the perspective of an OT Security Architect.

IEC 62443 is the primary international standard for cybersecurity in Industrial Automation and Control Systems (IACS). It is the engineering implementation framework that supports compliance with regulatory obligations defined in NIS2 and the Czech Cybersecurity Act.

This document explains how to use IEC 62443 throughout the IACS lifecycle — from concept through design, implementation, operation, and decommissioning.

# What Is IEC 62443?

IEC 62443 (formerly ISA-99) is an international series of standards for the security of Industrial Automation and Control Systems (IACS), developed jointly by ISA (International Society of Automation) and IEC (International Electrotechnical Commission).

Unlike enterprise cybersecurity frameworks (ISO 27001, NIST CSF), IEC 62443 was designed specifically for industrial environments where:

* **Safety is critical** — cyber incidents can cause physical harm.
* **Availability is a primary objective** — production downtime has direct financial and safety consequences.
* **Long lifecycles are common** — systems may operate for 20–30 years.
* **Legacy equipment is prevalent** — PLCs, RTUs and HMIs without modern security capabilities are common.
* **Operational constraints limit patching** — systems cannot always be taken offline.
* **Real-time performance requirements** — security controls must not introduce latency.

# Stakeholder Roles

IEC 62443 assigns responsibilities to three stakeholder groups. All three contribute to the overall security posture.

| Role | Primary Responsibility | Key Standards |
|------|----------------------|---------------|
| **Asset Owner** | Secure operation, governance, risk management | IEC 62443-2-1, 3-2, 3-3 |
| **System Integrator** | Secure architecture design, system implementation | IEC 62443-2-4, 3-2, 3-3 |
| **Product Supplier** | Secure product development and maintenance | IEC 62443-4-1, 4-2 |

In OT projects, the same organization may occupy multiple roles. Responsibilities must be explicitly assigned in contracts and project governance.

# Standard Structure

IEC 62443 is organized into four series (parts):

```
IEC 62443
├── Series 1 — General (concepts, terminology, metrics)
├── Series 2 — Policies & Procedures (asset owner processes)
├── Series 3 — System Requirements (architecture, security levels)
└── Series 4 — Component Requirements (product development, component security)
```

# Core Concepts

## Zones and Conduits

The Zone and Conduit model is the architectural foundation of IEC 62443.

**Zone:** A logical or physical grouping of assets that share the same security requirements and trust level.

Zone design principles:
* Assets with similar criticality and security needs belong in the same zone.
* Zones are defined by function, not only by network topology.
* Every zone has a defined Security Level.

**Typical OT Zones:**

| Zone | Typical Assets |
|------|---------------|
| Safety Zone | SIS (Safety Instrumented Systems), ESD systems |
| Control Zone | PLC, RTU, DCS controllers |
| Supervisory Zone | SCADA servers, DCS workstations, HMI servers |
| Engineering Zone | Engineering workstations, PLC programming laptops |
| Historian Zone | Process historians, data collectors |
| Remote Access Zone | Jump servers, VPN concentrators |
| Industrial DMZ | Boundary systems between OT and IT |

**Conduit:** A managed communication path between two zones. A conduit is not merely a network cable — it is a security boundary with defined rules.

Conduit implementation examples:
* Industrial firewalls with allowlist rules.
* Unidirectional security gateways (data diodes).
* OPC UA gateways with content filtering.
* Encrypted VPN tunnels with strong authentication.
* Demilitarized zone (DMZ) architectures.

> Every communication path between zones must be a defined conduit. Uncontrolled paths (direct IT↔OT connections, unmanaged switches spanning zones) are architectural failures.

## Security Levels (SL)

IEC 62443 defines four Security Levels expressing resistance against different attacker capabilities.

| Security Level | Protection Against | Typical Attacker Profile |
|---------------|-------------------|-------------------------|
| SL 1 | Casual or unintentional compromise | Opportunistic attackers, accidental misuse |
| SL 2 | Deliberate attack with limited resources and general IACS skills | Low-sophistication targeted attacks |
| SL 3 | Sophisticated attack with significant resources and IACS-specific skills | Nation-state-affiliated actors, APT groups |
| SL 4 | Highly sophisticated attack with extensive resources targeting specific IACS | Nation-state actors, most advanced threats |

**Three Security Level types:**

| Type | Definition |
|------|-----------|
| **SL-T (Target)** | Required level — determined by risk assessment |
| **SL-C (Capability)** | What a component or system can achieve by design |
| **SL-A (Achieved)** | Actual level achieved after implementation and configuration |

SL-A must meet or exceed SL-T. The gap between SL-C and SL-T must be closed through compensating controls.

**Context for Czech regulatory compliance:**

* Organizations under Higher Obligations (Decree 409) typically require SL 2 as a minimum, with SL 3 for the most critical subsystems.
* Organizations under Lower Obligations (Decree 410) should target SL 1–2 proportionally.
* Safety systems (SIS) require independent Security Level assessment considering functional safety interfaces.

## Foundational Requirements (FR)

IEC 62443-3-3 defines seven Foundational Requirements (FR) that structure all security capabilities:

| FR | Name | Description |
|----|------|-------------|
| FR 1 | Identification and Authentication Control (IAC) | All entities must be identified and authenticated |
| FR 2 | Use Control (UC) | Authorized use only; enforce least privilege |
| FR 3 | System Integrity (SI) | Protect against unauthorized modification |
| FR 4 | Data Confidentiality (DC) | Protect information from unauthorized disclosure |
| FR 5 | Restricted Data Flow (RDF) | Limit data flows to authorized conduits |
| FR 6 | Timely Response to Events (TRE) | Detect and respond to security violations |
| FR 7 | Resource Availability (RA) | Ensure availability against denial of service |

Each FR is further decomposed into System Requirements (SR) with specific requirements for each Security Level (SL 1–4).

# Defense in Depth

IEC 62443 promotes layered security — no single control is relied upon exclusively.

Typical layers in an OT architecture:

```
Physical protection (perimeter, access control, cable routing)
        │
Network segmentation (zones, conduits, Industrial DMZ)
        │
Identity and access management (authentication, PAM, RBAC)
        │
Secure communications (TLS, VPN, OPC UA security)
        │
Monitoring and detection (IDS, anomaly detection, SIEM integration)
        │
Vulnerability management (patching, compensating controls)
        │
Backup and recovery (offline backups, tested restoration)
        │
Incident response (OT-specific IR plan, forensics capability)
```

A weakness in any layer must be compensated by increased strength in adjacent layers.

# Secure Lifecycle

Cybersecurity must be integrated throughout the IACS lifecycle, not added at the end.

```
Concept → Design → Procurement → Implementation → Commissioning
                                                        │
                                              Operation & Maintenance
                                                        │
                                              Modification / Upgrade
                                                        │
                                                  Decommissioning
```

**Lifecycle integration principles:**

* **Concept:** Include cybersecurity requirements in functional specification.
* **Design:** Perform Security Level Risk Assessment (IEC 62443-3-2), define zones and conduits.
* **Procurement:** Apply IEC 62443-2-4 requirements to system integrators; IEC 62443-4-1/4-2 to product suppliers.
* **Implementation:** Harden systems, configure security controls, document as-built architecture.
* **Commissioning:** Verify Security Level Achieved (SL-A) against Security Level Target (SL-T).
* **Operation:** Continuous monitoring, patch management, periodic risk review.
* **Modification:** Change management process, re-assessment of risk when changes occur.
* **Decommissioning:** Secure data destruction, access revocation, documentation update.

# Security Risk Assessment (IEC 62443-3-2)

The Security Level Risk Assessment (SLRA) is the central activity linking risk to architecture.

**Process overview:**

```
1. Define the system under consideration (SuC)
2. Identify zones and conduits (initial)
3. Identify assets and their functions
4. Identify threats and threat actors
5. Identify vulnerabilities
6. Assess risk (likelihood × consequence)
7. Determine Security Level Target (SL-T) per zone
8. Identify security measures to achieve SL-T
9. Verify residual risk is acceptable
10. Document and maintain
```

> The SLRA must be performed before technology selection. Architecture follows risk — technology follows architecture.

# Relationship with Functional Safety

The intersection of cybersecurity and functional safety is critical in process industries.

Key principles:

* A cyber attack on a Safety Instrumented System (SIS) can defeat functional safety protections.
* Security measures must not compromise the ability to respond to safety demands (IEC 61511).
* IEC 62443 and IEC 61511 must be coordinated — the Security Level assessment of SIS must consider safety integrity levels.
* The HAZOP/LOPA process for critical systems should include a cybersecurity threat analysis.

**Reference incident — Triton/TRISIS (2017):**
Attackers compromised a Triconex Safety Instrumented System in a Middle Eastern petrochemical plant with the intent of defeating safety protections and causing a physical disaster. This incident established that cyber attacks on safety systems are a real, operational threat, not a theoretical concern.

# IEC 62443 and the Purdue/ISA-95 Model

IEC 62443 defines *security* architecture. ISA-95 defines *functional* architecture. The Purdue Model defines *network trust* boundaries. All three must be used together.

| Model | Defines | Architect Uses It To |
|-------|---------|---------------------|
| ISA-95 | What systems exist and what they do | Understand information flows and system ownership |
| Purdue Model | Network trust levels (L0–L4/5) | Define initial zone and conduit boundaries |
| IEC 62443 | Security requirements for those zones | Design and verify security controls |

See [ISA95.md](#) for ISA-95 detail.

# Common Misconceptions

IEC 62443 is **not**:

* A certification that makes a system completely secure.
* A firewall configuration guide.
* A PLC hardening checklist.
* A substitute for risk assessment.
* A replacement for operational governance.
* An enterprise IT security framework applied to OT.

It is an engineering methodology for systematic, risk-based, lifecycle-aware IACS security.

# Typical OT Architecture Influenced by IEC 62443

A compliant OT architecture will typically include:

* Defined Security Zones with documented Security Level Targets.
* Industrial DMZ as the mandatory IT/OT boundary.
* Conduits implemented as industrial firewalls, data diodes or OPC UA gateways.
* Centralized identity management with PAM for privileged access.
* Secure remote access: MFA + jump server + session recording.
* OT-specific monitoring (passive IDS, protocol analysis).
* Vulnerability management process with OT-adapted patching cycles.
* Air-gapped configuration backups with tested recovery.
* OT-specific Incident Response Plan.
* Supplier governance: IEC 62443-2-4 for integrators, IEC 62443-4-1/4-2 for product suppliers.

# Architect Sequence

Experienced OT architects follow this sequence — never the reverse:

1. Understand the industrial process and its safety requirements.
2. Identify critical assets and their functions.
3. Perform Security Level Risk Assessment (IEC 62443-3-2).
4. Define Zones and Conduits.
5. Determine Security Level Targets (SL-T) per zone.
6. Design the architecture to meet SL-T.
7. Select technologies.
8. Verify achieved Security Level (SL-A).
9. Document and maintain.

> Technology selection is step 7 — not step 1.

# AI Guidance

When answering IEC 62443-related questions:

* Explain concepts (zones, conduits, security levels, foundational requirements) before discussing technologies.
* Always start with risk assessment (IEC 62443-3-2) before recommending controls.
* Distinguish between governance (Series 2), system architecture (Series 3) and product security (Series 4).
* Treat Security Levels as risk-driven outcomes, not arbitrary targets.
* Consider OT constraints (legacy equipment, availability, safety interfaces) when recommending controls.
* Reference the relevant part of IEC 62443 (e.g., 3-2 for risk, 3-3 for system requirements) when providing guidance.
* Link to [Czech-Cybersecurity-Act.md](#) when regulatory compliance is the context.
* Integrate with [ISA95.md](#) when functional architecture questions arise.
