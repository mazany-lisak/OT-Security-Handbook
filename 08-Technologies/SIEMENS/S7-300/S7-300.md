# Purpose

This document provides a security architecture and **compensating-control** reference for the Siemens **SIMATIC S7-300** programmable logic controller, from the perspective of an OT Security Architect.

The S7-300 was, for two decades, one of the most widely deployed PLCs in the world — compact, modular, and the default choice across manufacturing, machine building and process automation. It is now **discontinued by Siemens** (end of manufacture; spare-part and support runout in progress across the range), yet enormous numbers remain in active service. Like all controllers of its generation it has **no meaningful native security**: unauthenticated S7comm, a mechanical key switch as the only access-level control, and application (block) protection that is a deterrent, not a defence. Its security posture is therefore, like [Modicon-Premium.md](../../Schneider%20Electric/Modicon%20Premium/Modicon-Premium.md), almost entirely a function of the **architecture around it** — the opposite emphasis to a modern controller such as [S7-1500.md](../S7-1500/S7-1500.md), whose document is organised around native hardening.

This document deliberately foregrounds containment, detection and migration rather than device hardening. Configuration procedures are excluded; consult the deployed STEP 7 and hardware documentation.

> **Historical grounding, not sensationalism.** The S7-300 is the family **Stuxnet** targeted (2010): the malware propagated via Windows engineering hosts and manipulated S7-300 CPUs controlling centrifuge frequency converters, reprogramming logic through unauthenticated S7comm while replaying recorded values to operators to conceal the manipulation. It remains the canonical case study for *why* unauthenticated engineering protocols plus compromised engineering workstations equal physical consequence, and it is the reason IEC 62443 and this handbook insist on engineering-workstation governance and process monitoring, not device trust. See [ICS-Kill-Chain.md](../../../07-Threat-Intelligence/ICS-Kill-Chain.md) and [MITRE-ATTACK-ICS.md](../../../07-Threat-Intelligence/MITRE-ATTACK-ICS.md).

> **Verification / lifecycle note.** Siemens has published defined end-of-life milestones for the S7-300 range (manufacture discontinued; spare parts and repair phased out over subsequent years, varying by module). Verify the exact CPU/module, firmware and current support status of deployed units against Siemens' product lifecycle and compatibility tools; treat this document as the *pattern*.

# 1 — Overview

## 1.1 Platform Role

The S7-300 is a rack/rail-mounted modular PLC: a CPU module plus signal, communication and function modules on a backplane, communicating over **MPI**, **PROFIBUS DP**, and — where an Ethernet **CP (Communication Processor)** module is fitted, or on later integrated-Ethernet CPU variants — **Industrial Ethernet** using **S7comm** (the "classic" S7 protocol, distinct from the S7comm-plus used by the S7-1500 family). It typically sits as the last programmable device before physical actuation, in machines, process skids and distributed plant sections.

## 1.2 CPU Range and Programming

* **CPU classes:** a broad range from compact/economy CPUs to performance/technology CPUs (with integrated counting, positioning or PID capability), plus **fail-safe CPUs (F-CPUs)** for SIL/PL-rated safety functions.
* **Programming:** **STEP 7 V5.x (Classic)**; later CPU variants are also programmable from **TIA Portal**. The toolchain in use determines what limited protection features are even reachable (section 4).
* **Networks:** MPI (engineering/small networks), PROFIBUS DP (fieldbus to distributed I/O and drives), and Ethernet via CP or integrated interface for S7comm, HMI/SCADA polling and (on some CPs) further protocols.

## 1.3 ISA-95 / Purdue Placement

* Controller at **Level 1**; its I/O, drives and field devices at **Level 0**.
* Programming (STEP 7/TIA Portal) from a **dedicated engineering workstation** in a separate engineering zone — never an operator or office machine. This is precisely the boundary Stuxnet crossed.
* No enterprise or direct remote reachability. Because the device authenticates nothing, its **zone boundary is its only real access control** ([Purdue-Model.md](../../../02-Standards/Purdue-Model.md)).

## 1.4 Security-Capability Reality

| Capability | Modern controller (e.g. S7-1500) | Siemens S7-300 |
|---|---|---|
| Control/engineering protocol authentication | Partial (UMAC, secure PG/PC on capable firmware) | **None** — S7comm accepts unauthenticated read/write/start/stop |
| Transport encryption | TLS/secure comms on capable CPUs | **None** |
| Firmware signing / secure boot | Yes on current CPUs | **No operator-exposed integrity** |
| Application protection | UMAC roles, know-how/copy protection | **Mechanical key switch + weak, historically bypassable block password** |
| Access control lists on the CPU | Yes | **No** (rely on network) |
| Audit logging on the device | Limited but present | **Effectively none** |

The consequence is identical to every legacy controller in this handbook: **every protection must come from the surrounding architecture.** The S7-300 is a textbook IEC 62443 *compensating-control* case ([Risk-Based-Security.md](../../../03-Architecture/Risk-Based-Security.md)).

# 2 — Security

## 2.1 Security Philosophy

For the S7-300, the question is never "how do we harden the CPU" — there is little to harden — but "how do we **contain, monitor and eventually replace** it." The device will execute whatever a reachable S7comm client tells it to, will not distinguish an authorised engineer from an attacker, and will not record that a program was changed. Security is a property of the network zone, the engineering workstation, and process-behaviour monitoring — plus a funded migration plan.

## 2.2 Threat Model

1. **Unauthenticated S7comm read/write/start/stop** — any host reachable to the CPU can read and write data blocks, and issue mode changes (RUN/STOP), with no credential. This is the dominant risk and the precise mechanism Stuxnet exploited to alter centrifuge control logic while replaying stale process values to the HMI.
2. **Unauthorised program upload/download** — S7comm permits full block transfer; a reachable attacker, or malware staged on the engineering workstation, can silently replace control logic (ATT&CK for ICS: *Program Download*, *Modify Controller Tasking*, *Spoof Reporting Message*; [MITRE-ATTACK-ICS.md](../../../07-Threat-Intelligence/MITRE-ATTACK-ICS.md)).
3. **Engineering-workstation compromise** — STEP 7/TIA Portal on a shared, Internet-capable or removable-media-exposed machine is the single point of failure for every S7-300 it can reach; this is the Stuxnet propagation vector and remains the highest-consequence path ([ICS-Kill-Chain.md](../../../07-Threat-Intelligence/ICS-Kill-Chain.md) → Engineering Access).
4. **Key-switch and physical bypass** — the RUN/STOP/MRES mechanical key switch is a genuine control only when its position is enforced and monitored; left in RUN-P (programming enabled) permanently, it offers nothing.
5. **Weak block/know-how protection** — legacy block-password protection on this generation is well documented as bypassable; treat it as friction against casual copying, never as access control.
6. **PROFIBUS/MPI exposure** — these fieldbuses are unauthenticated by design; bridging them onto Ethernet without a monitored, contained gateway extends an open trust domain.
7. **Known-vulnerability exploitation** — the S7-300/S7comm family has a substantial public advisory history (denial-of-service via malformed packets, authentication weaknesses, information disclosure); with many units unable to receive further firmware fixes, this is a *containment* problem, not a *patch* problem ([Vulnerability-Management.md](../../../09-Operations/Vulnerability-Management.md)).
8. **Denial of control** — malformed or excessive traffic can fault or stop an old CPU; at Level 1 this is a safety and availability concern, not merely an IT outage.
9. **Spare-part exhaustion** — with manufacture discontinued, an unplanned failure (or post-incident replacement need) may have no clean part supply; this operational risk compounds every other item on this list.

## 2.3 Defense in Depth Applied

| Layer | S7-300 specifics |
|---|---|
| Physical | Locked cabinet; key-switch discipline with monitored position; controlled MPI/programming-port access |
| Network | **The primary control** — tight L1 zone, S7comm-aware firewall or data diode, no flat OT network, PROFIBUS/MPI never bridged onto general Ethernet |
| Identity | Not on the device — enforced at the engineering workstation and the zone boundary |
| Application | Enable block/know-how protection where available; treat as deterrence only |
| Endpoint | Hardened, dedicated, removable-media-controlled engineering PC; allowlisting; no email/browsing |
| Monitoring | S7comm-aware OT IDS in front of the CPU — the only realistic way to see a write, a download or a mode change |
| Recovery | Offline, verified program backups + spares/replacement strategy |

The layered model is defined in [Defense-in-Depth.md](../../../03-Architecture/Defense-in-Depth.md); for the S7-300, network containment, engineering-workstation discipline and monitoring carry almost the entire weight.

## 2.4 IEC 62443 Alignment — as a Compensating-Control Case

The S7-300 cannot itself satisfy FR1 (identification/authentication) or FR3/FR4 (integrity/confidentiality) at the component level. The correct posture:

* Place it in a **low-capability zone** with an explicitly documented, *reduced* achieved Security Level.
* Raise the **zone's** effective protection through conduit controls (S7comm-aware firewall/diode), monitoring (FR6), and strict engineering-access governance — documented **compensating controls** with named risk acceptance ([Risk-Management-Principles.md](../../../00-Core/Risk-Management-Principles.md) → residual risk).
* Record the gap between Target SL and achieved SL as residual risk with a **migration commitment**, not a permanent accepted state.

# 3 — Compensating Controls

* **Zone isolation.** A dedicated S7-300/L1 zone; default-deny at the boundary; only the specific SCADA/HMI hosts that must poll it may reach it, only on the required S7comm/Modbus ports where present ([Network-Segmentation.md](../../../04-Network/Network-Segmentation.md), [Firewall-Design.md](../../../04-Network/Firewall-Design.md)).
* **S7comm-aware front-end enforcement.** An industrial firewall with **S7comm/DPI awareness** to constrain function codes (permit reads, tightly restrict/deny writes and mode-change functions from all but the engineering path) and rate-limit; where the flow is genuinely one-way (SCADA polling only), a **data diode** removes the write path entirely.
* **Engineering-access brokering.** Programming permitted only from the dedicated engineering workstation, only during change windows, ideally via a jump host — never a standing path ([Secure-Remote-Access.md](../../../04-Network/Secure-Remote-Access.md), [Privileged-Access-Management.md](../../../05-Identity/Privileged-Access-Management.md)).
* **Fieldbus containment.** Do not bridge PROFIBUS/MPI onto routable networks; terminate locally; inventory every gateway that does exist.
* **Behavioural monitoring.** OT IDS baselining normal S7comm so that any unexpected write, block transfer or mode change is an alert (section 6).
* **Key-switch discipline.** Set and monitor RUN/RUN-P/STOP position; treat unplanned changes as security events, not maintenance noise.

Compensating controls must be **documented, risk-accepted and periodically reviewed**, and paired with a migration plan; they reduce exposure — they do not make an unauthenticated 1990s-era protocol safe.

# 4 — Identity, Engineering and Protection

## 4.1 Identity Reality

There is no meaningful device-level identity. "Who may change this controller" is answered entirely by **who can reach it on the network** and **who controls the engineering workstation** — exactly the gap Stuxnet exploited. Design accordingly: named engineers, MFA and PAM on the engineering host ([Identity-Management.md](../../../05-Identity/Identity-Management.md), [MFA.md](../../../05-Identity/MFA.md)), and a zone boundary that admits only those hosts.

## 4.2 Access Levels and Block Protection

* The **mechanical key switch** (RUN / RUN-P / STOP, with MRES for reset on many CPUs) is the closest thing to a physical access control this generation offers — use it, and monitor its position as a security-relevant input, not just an operational one.
* **Block/know-how protection** (password-protecting individual blocks against viewing/modification) exists on later STEP 7 tool generations; enable it, but understand it has a well-documented history of weakness and recoverability on this platform generation. It deters casual copying and accidental modification; it does not withstand a determined actor with S7comm access.
* **Never** treat block protection as a substitute for network containment.

## 4.3 Engineering Governance

Every program change through change control ([Change-Management.md](../../../09-Operations/Change-Management.md)) with a **pre-change offline backup** and a known-good archived project (7). The engineering workstation is the crown jewel — and, per the Stuxnet precedent, the actual attack surface: dedicated, hardened, removable-media-controlled, ideally air-gapped except during controlled sessions, with STEP 7/TIA Portal licences and project sources under configuration management ([Configuration-Management.md](../../../09-Operations/Configuration-Management.md)).

# 5 — Communications Security

Per-protocol depth is owned by [OT-Protocols.md](../../../04-Network/OT-Protocols.md); S7-300-relevant points:

* **S7comm (classic)** — no authentication or encryption; supports read/write of data blocks and CPU start/stop/status functions with no credential. Confine to the L1 zone; constrain at an S7comm-aware DPI firewall; monitor every write and mode-change function.
* **MPI / PROFIBUS DP** — unauthenticated fieldbus; never route onto general networks; contain and document any Ethernet gateway.
* **Ethernet CP web/diagnostic services (where fitted)** — disable if unused; never expose beyond the zone; assume weak or no meaningful authentication on older CP firmware.
* **Time** — the CPU offers little to nothing usable; rely on network-side timestamping (IDS, firewall, SCADA) for forensic correlation ([Logging.md](../../../09-Operations/Logging.md)).

---

# 6 — Monitoring and Logging

Since the device itself barely logs, **detection lives in the network**. Deploy S7comm-aware OT IDS and forward to the SIEM ([SIEM.md](../../../09-Operations/SIEM.md), [Monitoring.md](../../../09-Operations/Monitoring.md)):

* Any **S7comm write** to the controller, and its source (baseline the legitimate SCADA/HMI poller and the engineering workstation; everything else is suspect).
* Any **program upload/download or CPU mode change (RUN/STOP)** — the reprogramming and disruption signal.
* **New or unexpected source hosts** talking to the CPU; new communication paths into the L1 zone.
* **Key-switch position changes**, where wired as a monitored input.
* Firewall denies at the zone boundary; unexpected PROFIBUS/MPI-gateway activity.
* Availability signals (CPU unreachable, comms faults) — at L1 this is a safety-relevant event, not just an IT ticket.

---

# 7 — Backup, Spares and Recovery

Recovery is the S7-300's genuine safety net, because on-device prevention is minimal:

* **Program backups.** Archived, version-controlled STEP 7/TIA project matching the running firmware, stored **offline** and periodically verified to open/compile ([Backup.md](../../../09-Operations/Backup.md)). A controller you cannot rebuild from a trusted project is unrecoverable after compromise or failure.
* **Firmware and tool custody.** Retain the exact STEP 7 version (or TIA Portal configuration) and any firmware needed to rebuild; capture hardware configuration, module layout and network settings.
* **Spares strategy.** With manufacture discontinued, actively maintain tested spare CPUs/modules or a documented rapid-replacement plan; spare scarcity directly lengthens RTO and is a genuine board-level risk for estates still dependent on this family.
* **Recovery testing.** Demonstrate restoration onto representative hardware ([Recovery-Testing.md](../../../09-Operations/Recovery-Testing.md)); recovery is complete only at a validated, safe process state.

---

# 8 — Migration and Decommissioning

The strategic control is **replacement**.

* **Target:** [S7-1500.md](../S7-1500/S7-1500.md) with **TIA Portal**, restoring native access-level management (UMAC), secure PG/PC communication, and OPC UA with security. Plan application conversion (STEP 7 Classic → TIA Portal is a supported but non-trivial migration), I/O re-architecture where module ranges differ, and full FAT/SAT ([OT-Lifecycle.md](../../../00-Core/OT-Lifecycle.md) → Modernization).
* **Prioritise** by consequence and exposure ([Risk-Based-Security.md](../../../03-Architecture/Risk-Based-Security.md)): Internet/remote-reachable and safety-relevant S7-300 units first; isolated, physically protected units can run longer under documented compensating controls.
* **Decommissioning:** retain or wipe application as required, remove from networks and asset inventory, revoke any associated access, dispose of media securely ([OT-Lifecycle.md](../../../00-Core/OT-Lifecycle.md) → Decommissioning). A forgotten, still-wired S7-300 on a flat network is a persistent, well-precedented liability.

# 9 — Best Practices and Common Mistakes

**Practices that pay off:** treat the network boundary as the real access control; S7comm-aware DPI or a data diode in front of the CPU; engineering only from a hardened, dedicated, media-controlled workstation in change windows; offline verified backups plus a genuine spares plan; OT IDS making every write and download visible; key-switch position monitored; a funded, consequence-prioritised migration to S7-1500; documented residual-risk acceptance.

**Common mistakes:**

* Assuming block protection secures the controller — it does not, against S7comm.
* S7-300 on a flat OT network reachable from IT or the Internet.
* STEP 7/TIA Portal installed on a shared or Internet-capable PC (the precise Stuxnet vector).
* PROFIBUS/MPI bridged onto routable networks and forgotten.
* Key switch left permanently in RUN-P "for convenience."
* No offline project backup; no spares plan — an unrecoverable single point of failure on a discontinued platform.
* No monitoring, so a reprogramming or mode change would never be seen.
* "It still works, leave it" — running a no-native-security, discontinued controller indefinitely with no migration plan and no documented risk acceptance.

# Architect Notes

The S7-300 carries a historical weight the other legacy controllers in this handbook do not: it is the device Stuxnet reprogrammed, and every element of that attack — unauthenticated S7comm, a compromised engineering workstation, silent logic replacement, spoofed process feedback — remains exactly as possible today on an unprotected S7-300 as it was in 2010. The defensible posture has not changed either: a tightly contained low-capability zone, network-side detection of every write and download, disciplined offline recovery, rigorous engineering-workstation governance, and a funded migration — with the gap between what the standard wants and what the device can do written down and accepted by someone with the authority to accept it.

# Decision Support

1. Is every S7-300 unit inventoried with CPU type, firmware, engineering-tool version and support status?
2. Is each in a dedicated L1 zone behind a default-deny, S7comm-aware boundary — or is it reachable flatly?
3. Are S7comm writes and program downloads to the CPU monitored and alertable?
4. Is programming restricted to a hardened, dedicated engineering workstation in change windows, with removable media controlled?
5. Are PROFIBUS/MPI paths contained and documented, not bridged?
6. Is there an offline, verified project backup and a tested spare/replacement plan?
7. Is the Target-vs-achieved SL gap documented as residual risk with named acceptance?
8. Is there a funded, consequence-prioritised migration plan to S7-1500 with a date?

# AI Guidance

When discussing Siemens S7-300:

* Lead with the reality that it has no native security; anchor recommendations in containment (zone/conduit, S7comm-aware DPI/diode), engineering-workstation governance and network-side monitoring — not device hardening.
* Use the Stuxnet precedent factually to explain *why* engineering-workstation compromise plus unauthenticated S7comm equals physical consequence — not as sensationalism, and never asserting technical details beyond well-established public reporting.
* Frame it explicitly as an IEC 62443 compensating-control and residual-risk case; insist on documented risk acceptance and a migration commitment.
* Highlight unauthenticated S7comm read/write/mode-change as the dominant threat; recommend OT IDS visibility of all three.
* Treat block/know-how protection and the key switch as deterrence/physical controls, never as access control substitutes.
* Recommend S7-1500 + TIA Portal as the strategic target; prioritise migration by consequence and exposure.
* Delegate protocol depth to OT-Protocols.md and recovery/lifecycle process to the 00-Core and 09-Operations layers; verify model/firmware/support specifics against current Siemens lifecycle and ProductCERT sources rather than asserting them.
