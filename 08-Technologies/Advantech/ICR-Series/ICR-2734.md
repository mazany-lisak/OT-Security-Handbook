> **What this document is.** A single, consolidated and source-verified knowledge base for the Advantech ICR-2734 industrial LTE router / secure edge gateway, its operating system ICR-OS, and its management platform WebAccess/DMP. It merges and supersedes the previous nine separate `ICR-2734-*.md` documents into one consistent reference, optimized for use as an **LLM skill and for Retrieval-Augmented Generation (RAG)**.
>
> **How it is written.** Each chapter is self-contained so that retrieved chunks remain meaningful in isolation. Hard product facts are clearly separated from architectural recommendations. Verified facts are drawn from Advantech's official engineering portal, hardware/configuration manuals, datasheets and the Secure S1 / WebAccess/DMP product pages (see *Sources*). Anything labeled a *recommendation*, *best practice* or *architecture* is design guidance aligned to IEC 62443, NIS2 and Zero Trust — not a vendor claim.
>
> **Accuracy caveat for the model.** Firmware behaviour, exact menu paths, supported algorithms and platform versions change between ICR-OS / WebAccess/DMP releases. When a user needs an authoritative, version-specific answer (a CLI command, a default value, a certification status), the model should direct them to the current Advantech Configuration Manual, Security Guidelines and Release Notes rather than inventing specifics.

## How to Use This Skill (Guidance for the LLM)

When answering questions about the ICR-2734, ICR-OS or WebAccess/DMP, the model should:

1. **Treat the ICR-2734 as an *Industrial Secure Edge Gateway*, not "just an LTE modem."** Its primary value is acting as the trusted security boundary between distributed OT assets and centralized operations.
2. **Lead with architecture and security**, then connectivity. Frame recommendations using IEC 62443 zones/conduits, the Purdue model, Zero Trust and Defense in Depth.
3. **Separate fact from recommendation.** State product capabilities as facts; state hardening/operational guidance as recommendations the customer must adapt to their risk assessment.
4. **Prefer centralized lifecycle management** (WebAccess/DMP, Configuration Profiles, Zero Touch Provisioning, controlled firmware) over per-device manual administration.
5. **Be precise about the S1 distinction.** The *standard* ICR-2734 is an open, highly configurable platform whose security depends on user setup. The *S1* variant is a hardened, IEC 62443-4-2 SL1-*aligned* build. "Aligned/compliant" is not the same as "formally certified" (see *S1 Secure Variant*).
6. **Avoid fabricating** CLI syntax, default credentials, exact spec values or certification claims. If unknown, say so and point to the official manual.
7. **Map controls to outcomes**: reduced attack surface, continuous trust verification, fast and validated recovery, configuration integrity, and operational resilience.

## Product Identity and Verified Facts

The ICR-2734 ("Libratum") is an **industrial LTE Cat.4 cellular router and IoT/edge gateway** in Advantech's well-balanced **ICR-2700/2800** family. It is the modern successor to the long-running **LR77 v2 Libratum** (form/fit compatible), entering mass production in 2022. It runs the Linux-based **ICR-OS** and is managed at scale by **WebAccess/DMP**.

### Regulatory and Standards Context

- **IEC 62443** — the core standard family for IACS security. The ICR-2734 is positioned as a *conduit/zone boundary* device; the **S1** variant is built in line with **62443-4-2 SL1** (component-level, Security Level 1) requirements.
- **NIS2 Directive (EU 2022/2555)** — raises cybersecurity and incident-reporting obligations for essential/important entities (energy, water, transport, manufacturing, etc.). The ICR-2734's role in segmentation, secure remote access, centralized logging and recovery supports NIS2 technical and organizational measures (risk management, access control, monitoring, business continuity).
- **RED Delegated Act 2022/30** — EU radio-equipment cybersecurity obligations (network protection, personal-data protection, fraud prevention). Advantech Czech cellular routers are being brought into alignment.
- **BSI IT Security Label** — the S1 program is also built in line with these rules.

> **Important accuracy note for the model:** Advantech states that S1 Routers are *produced in line with the recommendations* of IEC 62443-4-2 SL1 and the BSI IT Security Label, but that **formal certification is generally not claimed** for a given part number unless explicitly stated on its datasheet. The model must never assert formal certification; it should say "designed/aligned to SL1 requirements" and refer the user to the specific datasheet for certification status.

# 1. Overview

## 1.1 Architectural Role

Within an Industrial Automation and Control System (IACS), the ICR-2734 is the **secure communication gateway** between remote/distributed industrial assets and centralized operational infrastructure. It combines industrial networking, VPN connectivity, edge computing and centralized lifecycle management (via WebAccess/DMP) in one device.

Its architectural responsibilities are:

- Secure WAN connectivity (LTE, with failover)
- Secure remote maintenance/access
- Network segmentation enforcement
- VPN termination
- Edge computing / local data processing
- Device management and infrastructure monitoring

The router forms the **communication boundary** between distributed assets and central OT operations. Every connection entering the OT environment through it should be explicitly authorized.

## 1.2 Product Philosophy

The ICR-2734 should be regarded as an **Industrial Secure Edge Gateway**, not merely a cellular router. Treating it as a commodity modem is the single most common and damaging design mistake, because it discards the device's value as a security enforcement point and lifecycle anchor.

## 1.3 Purdue Model Placement

Typical deployment positions:

- **Level 3.5 — Industrial DMZ** (most common for enforcing zone/conduit boundaries)
- **Remote-site gateway**
- **Field-site gateway**

The router connects distributed assets **without exposing Level 0/Level 1 devices (sensors, PLCs/RTUs) directly** to external networks. Place the device so it *enforces* boundaries rather than bridging them.

## 1.4 Typical Architecture

```text
OT Operations / SOC / SIEM
        │
WebAccess/DMP (Management Plane)
        │  TLS / PKI / ZTP
Industrial DMZ (Purdue L3.5)
        │  WireGuard / IPsec / OpenVPN
ICR-2734 (Secure Edge Gateway)
        │
Engineering / Industrial VLAN(s)
        │
PLC • RTU • HMI • SCADA • Sensors
```

The router provides secure connectivity while **preserving** segmentation; WebAccess/DMP governs its lifecycle. Routers do **not** manage one another — the management plane controls the routers; the routers control connectivity.

## 1.5 Core Functions

LTE connectivity, WAN/SIM failover, VPN gateway, stateful firewall, NAT, routing, edge computing, device monitoring, and secure remote maintenance.

## 1.6 Communication & Protocols

LTE Cat.4 (3G/2G fallback), Ethernet, Modbus TCP and serial protocol conversion (via Router Apps), MQTT, SNMP, DHCP, NAT, VRRP, DynDNS, NTP. Industrial protocol conversion (e.g. IEC 60870-5-101↔104, DNP3, Modbus RTU/TCP) is delivered through Router Apps where applicable.

## 1.7 Practical Deployment Examples

- **Remote pumping station (water utility):** ICR-2734 + Modicon M580 + Geo SCADA, WireGuard VPN, WebAccess/DMP, central SIEM.
- **Manufacturing cell:** ICR-2734 + Siemens S7-1500 + plant SCADA, dedicated Engineering VLAN, industrial firewall, MFA-protected remote access.
- **Renewable energy site:** ICR-2734 + solar inverter/PLC, MQTT telemetry, WebAccess/DMP, centralized monitoring.

## 1.8 Decision Support (Overview)

1. Which Purdue zone hosts the router (L3.5 DMZ vs. remote-site gateway)?
2. Which VPN technology fits the security architecture (WireGuard preferred)?
3. How are administrative identities managed (central IdP, RADIUS/TACACS+)?
4. Is WebAccess/DMP available (SaaS or on-premises)?
5. Which Router Apps are genuinely required?
6. How does monitoring integrate with the SIEM?
7. Which Disaster Recovery objectives (RTO/RPO) apply?
8. Which configuration baseline is enforced?
9. Is the **S1** variant required for the site's compliance/risk profile?

# 2. Security

## 2.1 Security Philosophy

The ICR-2734 is the **primary security enforcement point** protecting remote industrial assets. For many distributed sites it is the *first and sometimes only* security control in front of the OT network — a compromised edge gateway can expose every downstream asset. Security must therefore be designed across the **entire device lifecycle**, not bolted on at deployment.

## 2.2 Security Objectives

A secure ICR-2734 deployment should prevent unauthorized remote access, protect industrial assets, minimize attack surface, enforce least privilege, protect administrative interfaces, support centralized lifecycle management, enable secure monitoring, and maintain operational availability.

## 2.3 Reference Security Architecture

```text
Remote Engineer
   │  MFA
Identity Provider (RADIUS / TACACS+ / central IdP)
   │
WireGuard / IPsec VPN
   │
Industrial DMZ (Purdue L3.5)
   │
ICR-2734  ── Default-Deny firewall, segmentation, logging
   │
Engineering VLAN
   │
PLC / HMI / RTU
```

Remote access must **terminate at the gateway** and be explicitly authorized before any traffic reaches industrial assets.

## 2.4 Defense in Depth (Layered Controls)

No single control should be relied upon. Recommended layers:

1. **Physical protection** — locked cabinet, protected cabling/SIM.
2. **Secure firmware / boot** — (Secure Boot + integrity check on S1).
3. **Identity verification** — named accounts, MFA, RBAC.
4. **VPN encryption** — WireGuard/IPsec with cryptographic credentials.
5. **Firewall policy** — Default Deny, stateful, explicit allow.
6. **Network segmentation** — management/engineering/PLC/SCADA VLANs + DMZ.
7. **Monitoring & logging** — centralized Syslog/SNMP/SIEM, continuous trust verification.

## 2.5 Native Security Capabilities (ICR-OS)

HTTPS, SSH, SFTP, stateful firewall, IP filtering, MAC filtering, WireGuard, IPsec, OpenVPN, GRE, RADIUS, TACACS+, Two-Factor Authentication, and SCEP (where supported). The web interface supports **HTTPS with TLS 1.2 or later**; legacy/cleartext services (Telnet, HTTP, FTP) should be disabled.

## 2.6 S1 Secure Variant (Hardened Build)

For higher-assurance and compliance-driven deployments, the **ICR-2734-S1** family runs the dedicated **ICR-OS S1** firmware branch and adds:

- **Secure Boot** and **system integrity checking** (a "trusted chain from production to operation").
- **Encrypted firmware images**.
- **Read-only filesystem**.
- **AIDE** (Advanced Intrusion Detection Environment) for file-integrity monitoring.
- **Strong password enforcement** (e.g. ≥12 characters across multiple character classes).
- **Blocked insecure services** (Telnet, FTP).
- **Restricted root access** (limited command set).
- **Strong cryptographic algorithms** (strength ≥128-bit; e.g. SHA-256+, RSA-3072+, Ed25519).
- **Scripts only via S1 Router Apps** (no arbitrary startup scripts).
- Optional **TPM 2.0** and use of **OTP memory** as hardware roots of trust.

Built in line with **IEC 62443-4-2 SL1** and **BSI IT Security Label** rules. The trade-off versus the standard platform is **reduced flexibility for higher assurance**. The **ICR-OS S1 branch is exclusive** to S1 hardware (it cannot be loaded onto standard units, and standard firmware cannot be loaded onto S1 units). The GUI is the same, distinguished by an **"S1" label** and **orange colour scheme**.

> **Model guidance:** Recommend the S1 variant when the site must align to IEC 62443-4-2 SL1 / BSI rules, when supply-chain/firmware integrity is a stated requirement, or when the customer wants a vendor-hardened baseline rather than self-hardening a standard unit. Always clarify "aligned/compliant," not "certified."

## 2.7 Administrative Interfaces

HTTPS-only; SSH only when operationally required; disable unused management services; restrict management access by source IP; use a separate management VLAN; enforce administrative session timeouts and a strong password policy. **Management interfaces must never be Internet-accessible.**

## 2.8 VPN Security (Hierarchy)

| Priority | Technology | Notes |
|---|---|---|
| Preferred | **WireGuard** | Modern, fast, small attack surface, key-based |
| Alternative | **IPsec (IKEv2)** | Where interoperability with existing infrastructure is required |
| Legacy | **OpenVPN** | Compatibility scenarios |
| Restricted | **GRE** | Only combined with additional security (e.g. inside IPsec) |

VPN authentication should rely on **cryptographic credentials** rather than passwords wherever possible. The ICR-2734 supports all of the above, letting the organization match its security architecture.

## 2.9 Firewall Design

Default Deny; explicit Allow; stateful inspection; separate inbound/outbound policies; restrict management and VPN traffic; block unnecessary Internet access. Policies must be documented and periodically reviewed.

## 2.10 Network Segmentation

Recommended zones: Management VLAN, Engineering VLAN, PLC VLAN, SCADA VLAN, and an Industrial DMZ. The router should **enforce** boundaries, never bridge them. Avoid flat Layer 2 OT networks and restrict routing between security zones.

## 2.11 Common Security Mistakes

Direct Internet administration; shared admin accounts; missing MFA; flat OT networks; unrestricted VPN access; no firmware lifecycle; no centralized monitoring; no configuration baseline; no backup validation.

## 2.12 Decision Support (Security)

Which identities administer the router? Is MFA mandatory? Which VPN technology? Which management interfaces are required? Which firewall policy is enforced? How is firmware maintained? Which events are forwarded to the SIEM? How is configuration compliance verified? Is the S1 variant required?

# 3. Hardening

## 3.1 Hardening Philosophy

The purpose of hardening is **not to add security features — it is to remove unnecessary risk.** Every enabled service, user, open port, protocol and installed app increases the attack surface. A secure router exposes only the minimum functionality required for the operational objective. **Configuration simplicity is itself a security advantage.** Hardening is a *continuous lifecycle process*, not a one-time deployment checklist.

## 3.2 Hardening Lifecycle

```text
Factory Default → Initial Configuration → Security Baseline → Validation →
Production Deployment → Continuous Monitoring → Periodic Review → Recovery Testing
```

## 3.3 Phased Hardening Methodology

**Phase 1 — Physical security.** Install in a locked control cabinet; protect power, Ethernet, LTE cabling and SIM slots; record serial numbers; label assets per inventory. Physical compromise can bypass logical controls.

**Phase 2 — Firmware baseline.** Deploy only approved, supported firmware with current security fixes; maintain a firmware inventory; validate and stage updates; keep rollback images; document changes.

**Phase 3 — Administrative access.** HTTPS only (disable HTTP); SSH only when required (disable Telnet and FTP); restrict management to trusted networks; configure automatic session timeout. The web interface supports TLS 1.2+; disable legacy protocols.

**Phase 4 — Identity hardening.** Named admin accounts; strong password policy; MFA; least privilege; role separation; periodic credential review; immediate removal of unused accounts. **Never use shared admin accounts.**

**Phase 5 — Service hardening.** Review every enabled service and disable anything not operationally justified — Telnet, FTP, unused VPN/routing protocols, unused serial interfaces, unused Router Apps, test services. Every enabled service must have a documented justification.

**Phase 6 — Network hardening.** Dedicated management VLAN; engineering VLAN; Industrial DMZ; stateful firewall with Default Deny; restrict inbound management and outbound traffic; disable unnecessary port forwarding.

**Phase 7 — VPN hardening.** Prefer WireGuard, then IPsec (IKEv2), then OpenVPN (legacy). Use cryptographic keys, restrict peer definitions to approved endpoints, enforce MFA for admins, and apply firewall enforcement. VPN must terminate before access to industrial assets.

**Phase 8 — Certificate management.** Replace default certificates; deploy CA-issued certificates where practical; monitor expiry; rotate per policy; protect private keys. (See *Certificates*.)

**Phase 9 — Router Apps.** Install only required apps; validate origin; maintain a software inventory; test before deployment; remove unused apps; review permissions. Treat Router Apps as production software.

**Phase 10 — Logging.** Enable centralized logging and forward to a Syslog server (authentication, failed logins, configuration changes, firmware updates, VPN establishment, reboots, administrative actions).

**Phase 11 — Monitoring.** Monitor availability, VPN status, firewall events, firmware compliance, configuration drift, LTE quality, CPU/memory, and Router App health — operational *and* security events.

**Phase 12 — Configuration integrity.** Verify the approved Configuration Profile, firmware version, firewall rules, VPN configuration, Router Apps and user accounts. Drift should generate alerts.

**Phase 13 — Backup.** Regularly back up configuration, VPN settings, user configuration, certificates, Router Apps and Configuration Profiles; validate via Recovery Testing.

**Phase 14 — Operational maintenance.** Periodically review firmware, passwords, certificates, configuration, users, firewall, VPN and inventory reconciliation.

## 3.4 Hardening Validation Checklist

- [ ] Physical access restricted
- [ ] Approved firmware installed
- [ ] Default credentials removed
- [ ] HTTPS enabled; HTTP / Telnet / FTP disabled
- [ ] SSH access restricted (or disabled)
- [ ] Firewall follows Default Deny
- [ ] VPN operational and key-based
- [ ] Configuration backed up and validated
- [ ] Logs forwarded to Syslog/SIEM
- [ ] Monitoring active
- [ ] Recovery procedures tested

## 3.5 Common Hardening Mistakes

Leaving factory defaults; shared admin accounts; Internet-exposed management; unnecessary Router Apps; outdated firmware; ignoring configuration drift; no centralized monitoring; unvalidated backups.

# 4. Best Practices

## 4.1 Design Philosophy

Never view the ICR-2734 as "just an LTE router." Treat it as an Industrial Secure Edge Gateway that establishes the security boundary, enforces secure communications, enables secure remote maintenance, supports centralized lifecycle management, and protects downstream OT assets.

## 4.2 Architectural Principles

1. Identity before connectivity.
2. Secure by default.
3. Least privilege.
4. Defense in Depth.
5. Configuration as Code.
6. Centralized lifecycle management.
7. Continuous monitoring.
8. Recovery by design.

## 4.3 Reference Architecture

```text
SOC / SIEM ── Syslog / API ── WebAccess/DMP (Management Plane)
        │ TLS / PKI / ZTP
   ICR-2734 (Secure Edge Gateway)
        │ WireGuard VPN
   Engineering VLAN
        │
   PLC • RTU • HMI • SCADA
```

The router protects the OT environment; WebAccess/DMP governs its lifecycle.

## 4.4 Network Architecture

Deploy at the OT edge; separate management, engineering and production networks; use Industrial DMZs for external connectivity; avoid flat Layer 2; restrict inter-zone routing; apply Default Deny; never expose PLCs directly to WAN/Internet.

## 4.5 Configuration as Controlled Asset

Use standardized Configuration Profiles; version control; peer review; formal change management; drift detection; baseline verification. Minimize manual configuration.

## 4.6 Firmware Lifecycle (Best-Practice Flow)

```text
Vendor Release → Laboratory Testing → Approval → Controlled Deployment → Validation → Production
```

Never deploy untested firmware to production. Manage centrally through WebAccess/DMP where possible. (See *Firmware* and *Patch Management*.)

## 4.7 Router Apps Governance

Install only operationally required apps; maintain an inventory; test before deployment; remove unused apps; review permissions; apply software lifecycle governance. Each app increases attack surface.

## 4.8 Common Design Mistakes

Treating the router as a commodity modem; flat OT networks; shared admin accounts; manual configuration; missing MFA; ignoring Router Apps governance; no centralized monitoring; no recovery testing; no lifecycle documentation.

## 4.9 Production-Readiness Checklist

Identity governance implemented; segmentation complete; VPN architecture approved; MFA enforced; Configuration Profiles assigned; firmware approved; monitoring operational; logs reaching the SIEM; backups validated; recovery procedures tested.

## 4.10 Architect Notes

The greatest value of the ICR-2734 is not connectivity — it is enabling secure remote operations, centralized governance, operational resilience, standardized deployments and reduced operational risk. The router should become the *trusted security boundary* between distributed industrial assets and centralized OT operations.

# 5. Identity and Access Protection

## 5.1 Principles

Identity comes before connectivity. Administrative identities should be governed through **organizational processes and a central identity source**, not device-local ad-hoc accounts. The goal is verifiable, least-privilege, individually attributable access.

## 5.2 Recommended Controls

- **Named (individual) administrator accounts** — never shared accounts.
- **Multi-Factor Authentication (MFA / 2FA)** for privileged access.
- **Role-Based Access Control (RBAC)** with role separation.
- **Least Privilege** — minimum rights for the task.
- **Centralized authentication** via **RADIUS** or **TACACS+** where available.
- **Regular access reviews** and **immediate removal of obsolete accounts**.
- **Privileged Access Management (PAM)** for high-risk operations.
- **Strong password policy** (S1 enforces ≥12 chars, multiple classes).

## 5.3 Authentication Architecture (ICR-OS)

ICR-OS supports named local accounts, RADIUS and TACACS+ integration, and Two-Factor Authentication. Prefer centralized identity (RADIUS/TACACS+ tied to the organization's IdP) so that joiner/mover/leaver processes, MFA and audit are managed centrally. WebAccess/DMP additionally provides platform-level RBAC, 2FA, fine-tuned permissions and multi-tenancy for the *management plane*.

## 5.4 Secure Remote Access Identity

Remote maintenance should require MFA, a VPN, named identities, RBAC, session logging, time-limited access, and an approval workflow where applicable. A permanent VPN tunnel must **not** imply permanent engineering access — separate connectivity from authorization.

## 5.5 Management Plane = Tier-0

WebAccess/DMP (and any central IdP/PKI it relies on) should be treated as a **Tier-0 administrative system** within the OT environment, with the strongest identity controls, because compromise of the management plane can affect every managed router.

## 5.6 Decision Support (Identity)

Which identities administer the router? Is MFA mandatory? Is authentication centralized (RADIUS/TACACS+)? How are privileged operations approved and logged? How are accounts reviewed and de-provisioned?

# 6. Certificates

## 6.1 Role of PKI

Certificates underpin device trust, mutual authentication and secure transport (HTTPS/TLS, VPN, WebAccess/DMP registration). Trusted certificates improve resistance to impersonation and man-in-the-middle attacks. The ICR-2734 supports replacing the default HTTPS certificate with a **CA-signed certificate**, and (where supported) **SCEP** for automated enrolment.

## 6.2 Certificate Inventory to Manage

- **HTTPS / web-interface certificate** (replace the default).
- **CA certificates / trust anchors**.
- **Device (identity) certificates** used for mutual authentication and ZTP.
- **VPN certificates** (where IPsec/OpenVPN use PKI).

## 6.3 Recommended Practices

- **Replace all default certificates** before production.
- Deploy **CA-issued certificates** wherever practical (internal or managed PKI).
- **Monitor expiration** and alert *before* expiry (a Medium-severity monitoring event).
- **Rotate** certificates per organizational policy.
- **Protect private keys** — store securely per PKI policy; never include unprotected private keys in plain backups.
- Use **strong algorithms** (on S1: strength ≥128-bit, e.g. SHA-256+, RSA-3072+, Ed25519).
- Centralize certificate lifecycle through PKI and, where supported, **SCEP** and **WebAccess/DMP** certificate management.

## 6.4 Certificates in Backup & Recovery

Back up HTTPS, CA, device certificates and trust anchors. During recovery and after a cybersecurity incident, **rotate credentials and certificates**. Treat backed-up cryptographic material as sensitive (encrypt at rest, restrict access). (See *Backup and Recovery*.)

## 6.5 Zero Touch Provisioning Trust

ZTP relies on **PKI-based mutual authentication** and TLS-protected communication so each router has a unique cryptographic identity. Provisioning should never rely solely on serial numbers or shared credentials. (See *Zero Touch Provisioning*.)

# 7. Firmware

## 7.1 Firmware as a Governed Asset

ICR-OS firmware is the platform's trust foundation. Firmware management must be **controlled, inventoried and recoverable** — never ad hoc. The S1 branch adds encrypted images, Secure Boot and integrity checking so that only trusted firmware runs.

## 7.2 Recommended Firmware Practices

- Maintain an **approved firmware baseline** and a **firmware inventory** (version, release notes, rollback image) kept **separately** from configuration backups.
- **Validate firmware** before deployment; **test in a staging/lab environment**.
- Use **controlled upgrades** with **Change Management** and a **rollback plan**.
- **Validate after update** (see *Recovery Validation*).
- Manage centrally through **WebAccess/DMP** (deployment scheduling, compliance reporting, rollback) wherever possible.
- For **S1**, use only the **ICR-OS S1 firmware branch** (it is exclusive to S1 hardware; standard firmware cannot be loaded onto S1 units and vice versa).

## 7.3 Firmware Lifecycle Flow

```text
Vendor Release → Laboratory Testing → Approval → Controlled Deployment → Validation → Production
```

> The model should keep *Firmware* (the asset and its baseline/lifecycle) distinct from *Patch Management* (the operational process of evaluating, scheduling and applying updates). They are complementary — see *Patch Management*.

## 7.4 Common Firmware Mistakes

Deploying untested firmware to production; no firmware inventory; ignoring firmware versions in backups; no rollback image; no change management; skipping post-update validation.

# 8. Syslog

## 8.1 Purpose

Forward router events to a **centralized Syslog server** so logs survive device loss and serve as the authoritative audit trail for incident response and forensics. Syslog is the backbone of OT log centralization for the ICR-2734.

## 8.2 Recommended Syslog Events

- Administrator authentication (success)
- Failed logins / authentication failures
- Configuration changes
- Firmware updates / installation
- Router reboot
- VPN events (establishment / failure)
- Firewall events
- Service failures and system errors
- Provisioning events (initial registration, profile deployment)

## 8.3 Syslog Best Practices

- Use a **dedicated, reachable central Syslog collector**; design so logging **survives WAN outages** where feasible (local buffering, store-and-forward).
- Forward to the **SIEM** for correlation with firewall, Active Directory, VPN and SCADA logs — this materially improves detection.
- Protect log transport and storage; retain per organizational/regulatory policy (NIS2 may impose retention and reporting obligations).
- Treat Syslog as the **authoritative audit trail**; ensure time is synchronized (see *SNMP/Diagnostics* and NTP) so events correlate accurately.

# 9. SNMP

## 9.1 Purpose

SNMP provides standardized infrastructure monitoring of the ICR-2734 (health, interfaces, cellular status, environment) and integrates with NMS/monitoring platforms.

## 9.2 Recommended SNMP Configuration

- **Use SNMPv3 only** — with **authentication** and **privacy (encryption)** enabled.
- **Avoid SNMPv1/SNMPv2c** in production (cleartext community strings) unless strictly required for legacy interoperability, and then only on a restricted management network.
- Restrict SNMP to the **management network/VLAN** and a **central monitoring server**.

## 9.3 What to Monitor via SNMP

Using standard and vendor-specific MIBs:

- System information and hardware identifiers
- Interface statistics / throughput
- LTE / mobile connectivity status and signal quality
- Temperature and supply voltage
- GPS location (on supported models)
- Digital I/O status

## 9.4 SNMP Traps and the Event Engine

The ICR-OS **Event Engine** can send **SNMP traps** as automated responses to operational events (e.g. WAN failure, temperature threshold). Combine SNMP polling (state) with traps (events) and forward both into centralized monitoring/SIEM. (See *Monitoring* and *ICR-OS → Event Engine*.)

# 11. Monitoring

## 11.1 Monitoring Philosophy

Monitoring is **not** about confirming a router is online — it is **continuous trust verification**. The strategy should answer: *Is the router operational? Secure? Healthy? Compliant? Has anything changed?* Continuous trust verification is more valuable than simple availability checks, and should reduce **MTTD** (Mean Time to Detect) and **MTTR** (Mean Time to Respond).

## 11.2 Six Monitoring Domains

**Availability** — online status, power state, reboots, LTE registration, WAN availability, Ethernet link, VPN connectivity. *Loss of communication triggers immediate investigation.*

**Performance** — CPU, memory, storage, interface throughput, LTE bandwidth, packet loss, latency, Router App resource consumption. *Degradation often precedes failure.*

**Network** — routing-table changes, WAN/SIM failover, LTE signal quality (RSSI/RSRP/RSRQ/SINR), IP changes, DNS resolution, NTP sync, VPN tunnel status.

**Security** — admin logins, failed authentication, MFA failures, configuration changes, firewall-policy modifications, VPN establishment/failure, firmware upgrades, service enable/disable, Router App install/remove. *Detects both malicious activity and drift.*

**Configuration Compliance** — approved Configuration Profile, firmware version, Router Apps inventory, user accounts, firewall/VPN configuration, certificate status, time sync. *Drift generates alerts and triggers review.*

**Hardware Health** — temperature, supply voltage, watchdog events, modem status, SIM status, storage health.

## 11.3 Monitoring Architecture

```text
SIEM / SOC ── Syslog / SNMP / API ── WebAccess/DMP ── ICR-2734
                                                       │
                                          VPN │ Firewall │ LTE │ Router Apps
                                                       │
                                              PLC / SCADA / RTU
```

Collect data **centrally**, not independently at each remote site.

## 11.4 Alert Prioritization

| Severity | Examples |
|---|---|
| **Critical** | Device offline; VPN unavailable; firmware integrity failure; unauthorized configuration change; multiple authentication failures |
| **High** | Configuration drift; WAN failover; SIM failover; router reboot; service failure |
| **Medium** | High CPU; low LTE signal; storage threshold; certificate approaching expiry |
| **Low** | Informational status changes; scheduled firmware updates; successful configuration deployment |

Align alerts with the organization's Incident Response procedures.

## 11.5 Key Performance Indicators

Device availability; VPN availability; configuration compliance; firmware compliance; MTTD; MTTR; authentication-failure rate; configuration-drift rate; recovery success rate.

## 11.6 Centralized Monitoring (WebAccess/DMP)

WebAccess/DMP centralizes device inventory, device health, VPN status, firmware compliance, Configuration Profiles, alerts, dashboards, **Digital Twin**, geographical mapping and remote diagnostics. Centralize operational monitoring here where possible, and feed security-relevant telemetry into the SIEM.

## 11.7 Common Monitoring Mistakes

Monitoring only ICMP/ping availability; ignoring configuration changes; no centralized logging; SNMPv1 in production; no alert prioritization; ignoring firmware compliance; no SIEM integration; not monitoring Router Apps.

## 11.8 Decision Support (Monitoring)

Which events are operationally critical? Which logs must be retained? Which metrics feed the SIEM? Which KPIs are measured? Which events need immediate response? How is drift detected? How does monitoring survive WAN outages? How does monitoring support Disaster Recovery?

# 12. Logging

## 12.1 Logging Strategy

Logging supports auditing, incident response and forensic investigation. The router generates local logs (System/Kernel/Reboot) and should **forward events centrally** (Syslog → SIEM). *Logging* (what events to capture and retain) and *Syslog* (the transport/centralization mechanism) are complementary; see *Syslog*.

## 12.2 Recommended Log Events

Administrator login; failed authentication; VPN establishment; configuration modifications; firmware installation; router reboot; network failures; system events; provisioning and identity-verification events; administrative actions.

## 12.3 Logging Best Practices

- Forward to **centralized Syslog**; correlate in the **SIEM**.
- Retain logs per **organizational and regulatory policy** (NIS2 reporting/retention).
- Ensure **time synchronization** for accurate correlation.
- Protect log integrity and access; treat logs as security assets.
- Use logs to feed **KPIs** (auth-failure rate, drift rate) and **Incident Response**.

# 13. Backup and Recovery

## 13.1 Backup Philosophy

A backup is **not** merely a copy of configuration files — it is the information required to **reconstruct a trusted operational state**. Recovery should restore identity, configuration, firmware, security, connectivity, monitoring and compliance. The objective is **operational trust**, not just connectivity. *A router that forwards packets is not "recovered" until it is also trusted, compliant, monitored, on approved firmware and configuration, and fully integrated.*

## 13.2 Recovery Objectives

Rapid restoration; configuration integrity; identity preservation; firmware consistency; secure communications; compliance validation; operational resilience. Recovery must be **repeatable and documented**.

## 13.3 Backup Scope

| Component | What to protect |
|---|---|
| **Configuration** | Router config, interfaces, routing, firewall rules, VLANs, DNS, NTP |
| **Firmware** | Approved version, repository, release notes, rollback image (kept separate from config) |
| **VPN** | WireGuard / IPsec / OpenVPN config, tunnel & peer definitions, routing |
| **Certificates** | HTTPS, CA, device certs, trust anchors (private keys secured per PKI policy) |
| **Router Apps** | Installed apps, versions, configuration, dependencies, custom apps, startup config |
| **Identity** | Serial number, asset ID, hostname, WebAccess/DMP registration, assigned Configuration Profile, inventory record |

## 13.4 Backup Frequency

| Component | Recommended frequency |
|---|---|
| Configuration | After every approved change |
| Firmware inventory | After updates |
| Router Apps | After changes |
| Certificates | After renewal |
| Inventory | Continuous |
| Configuration Profiles | After modification |

Any approved configuration change should **trigger a new backup**.

## 13.5 Backup Storage & Security

Use multiple independent locations (secure backup repository, configuration-management system, offline backup, DR repository, encrypted archive). Protect with encryption at rest and in transit, access control, RBAC, MFA, integrity verification and audit logging. **Backups are sensitive security assets.**

## 13.6 Recovery Scenarios

**Scenario 1 — Hardware failure:** install replacement → approved firmware → restore configuration → restore VPN → restore certificates → restore Router Apps → register with WebAccess/DMP → validate. With a pre-assigned Configuration Profile + ZTP, the replacement should come up with minimal manual work.

**Scenario 2 — Configuration corruption:** verify firmware integrity → restore approved configuration → validate routing → validate firewall → validate VPN → resume monitoring. Confirm integrity before returning to production.

**Scenario 3 — Cybersecurity incident:** isolate device → preserve forensic evidence → factory reset if required → install approved firmware → restore trusted configuration → **rotate credentials and certificates** → re-register with WebAccess/DMP if needed → validate secure operation → return to production **only after approval**. The router supports standard, factory and emergency factory-reset procedures; **always back up before a reset** where possible.

## 13.7 Recovery Validation

Recovery is complete only after verifying: firmware version, configuration checksum, VPN connectivity, firewall policy, user accounts, certificates, Router Apps, monitoring, logging and Configuration-Profile assignment.

## 13.8 WebAccess/DMP-Accelerated Recovery

Configuration Profiles, Zero Touch Provisioning, device inventory, firmware management, configuration deployment, centralized monitoring and Digital Twin dramatically simplify recovery. Assigning the existing Configuration Profile to a replacement enables rapid restoration of the desired operational state. **WebAccess/DMP does not replace backups** — keep independent, encrypted backups regardless.

## 13.9 Recovery Testing

Test periodically: configuration restore, hardware replacement, firmware rollback, VPN recovery, Router Apps recovery, WebAccess/DMP registration, monitoring validation. Testing must demonstrate that documented procedures actually work.

## 13.10 Common Backup Mistakes

Backing up only configuration; ignoring firmware versions; not protecting private keys; not backing up Router Apps; no backup encryption; no recovery testing; no version control; assuming WebAccess/DMP replaces backups.

## 13.11 Decision Support (Backup & Recovery)

Are all critical configuration elements included? Can a replacement be commissioned without manual reconfiguration? Are firmware versions controlled and recoverable? Are certificates/keys protected? Are Router Apps included? Is recovery tested and documented? Does recovery restore a *trusted operational state*?

# 14. Patch Management

## 14.1 Why Patch Management Is Distinct

*Firmware* (Chapter 7) defines the trusted baseline and its lifecycle. **Patch Management is the operational discipline** of continuously discovering, assessing, scheduling, deploying and verifying updates (ICR-OS firmware **and** Router Apps) across the fleet — a recurring process, not a one-off deployment. It is a core NIS2 and IEC 62443 expectation.

## 14.2 Patch Management Process (Recommended)

1. **Discover** — monitor Advantech release notes / security advisories for ICR-OS, the **ICR-OS S1 branch**, and Router Apps; subscribe to vendor channels.
2. **Assess** — evaluate relevance/severity against the deployed baseline and exposure; determine urgency.
3. **Test** — validate in a **staging/lab** environment before production.
4. **Approve** — via formal **Change Management**.
5. **Schedule** — define maintenance windows; consider operational/safety impact and **rollback** readiness.
6. **Deploy** — roll out in controlled waves, centrally via **WebAccess/DMP** (deployment scheduling, compliance reporting); **update incrementally** where the vendor requires it (e.g. WebAccess/DMP supports updates only between consecutive versions — do not skip version levels).
7. **Verify** — confirm version compliance, integrity and correct operation (see *Recovery Validation*); update inventory.
8. **Document** — record what was applied, where, when and by whom.

## 14.3 Patch Management Best Practices

- Maintain a **firmware/app version inventory** and **firmware-compliance KPI**; alert on mismatch (a Critical/High monitoring event).
- Keep a **rollback image** and tested rollback procedure for every production version.
- Always run **supported firmware with current security fixes**; avoid skipping intermediate versions on the management platform.
- Treat **Router Apps** updates with the same Change Management rigor as firmware.
- For **S1**, only the ICR-OS S1 branch is applicable; standard firmware cannot be applied to S1 devices.
- Coordinate patch windows with **High Availability** (Chapter 15) to minimize downtime (e.g. patch standby/secondary paths first).

## 14.4 Common Patch Management Mistakes

No advisory monitoring; patching production without staging; skipping incremental platform updates; no rollback plan; ignoring Router App updates; no version inventory; no post-patch validation.

# 15. High Availability

## 15.1 Principle

Availability should **complement, not replace, security**. Apply HA where operational requirements (RTO/RPO, criticality) justify the added complexity. The ICR-2734 provides several native resilience features; site-level HA combines these with redundant infrastructure.

## 15.2 Device-Level Resilience (ICR-2734 features)

- **Dual SIM** switching/failover between operators/networks.
- **WAN failover** and **backup routes**.
- **VRRP** for gateway redundancy.
- **Load balancing** across links (where configured).
- **Backup real-time clock**, **hardware watchdog**, **automatic restart** on connection loss, and **Sleep/Low-Power mode** for power-constrained sites.

## 15.3 Site / Architecture-Level HA (recommendations)

- **Redundant VPN hubs** (multiple concentrators / endpoints).
- **Independent management infrastructure** (resilient WebAccess/DMP deployment; for on-premises, plan DR of the platform itself).
- **Redundant monitoring** so visibility survives a single failure.
- Pair HA with **patch sequencing** so updates do not take down all paths simultaneously.

## 15.4 Management-Plane HA

If WebAccess/DMP is on-premises, include it in **Disaster Recovery**: protect platform availability, device inventory, Configuration Profiles, firmware repository, VPN orchestration, user identities and audit history. SaaS deployment shifts platform availability to the provider but the customer still owns device-side backup/recovery.

## 15.5 Decision Support (HA)

What are the site RTO/RPO and criticality? Is dual-SIM/WAN failover sufficient, or are redundant VPN hubs required? Is the management plane resilient? Does the monitoring stack survive single failures? How do HA and patch windows interact?

# 16. ICR-OS

## 16.1 What ICR-OS Is

ICR-OS is the **Linux-based industrial edge operating system** powering Advantech cellular routers including the ICR-2734. It is **not closed firmware** — it is a secure edge-computing platform combining networking, cybersecurity, automation and application hosting, with a web GUI and an open environment for **Router Apps** and custom development. It is available in a **standard** version and a hardened **Secure S1** version.

## 16.2 Platform Architecture

```text
WebAccess/DMP
   │ Management APIs
ICR-OS  ── Networking · VPN · Firewall · Authentication · Logging · Router Apps · Linux services
   │
Hardware platform: Ethernet · LTE · USB · Serial · GPIO (selected models)
```

A modular architecture separates platform services from customer applications, leaving the firmware core untouched and stable while Router Apps extend functionality.

## 16.3 Core Capabilities

Industrial networking, routing, VPN services, stateful firewall, NAT, VLAN, DHCP, SNMP, QoS, dynamic routing (where supported), logging, diagnostics, event automation and firmware management.

## 16.4 Networking

IPv4, IPv6 dual stack, VLAN, NAT/PAT, port forwarding, VRRP, DHCP, DNS proxy, QoS, backup routes, Ethernet bridging, load balancing, and dynamic routing (RIP, OSPF, BGP via Router Apps where supported). Design per IEC 62443 zones and conduits.

## 16.5 VPN Services

WireGuard (preferred) → IPsec (IKEv2) → OpenVPN, plus GRE, L2TP and DMVPN (where supported). Select per the organization's security architecture.

## 16.6 Router Apps

Router Apps extend ICR-OS and are downloadable/administrable through the web GUI. Examples: Modbus conversion, **IEC 60870-5-101↔104** conversion, VPN modules, dynamic-routing (OSPF/BGP/RIP) options, MQTT/IoT-platform connectors (Azure, Cumulocity), **Node-RED** nodes, **Zabbix** monitoring integration, sleep-mode configuration, HTTP authentication, Layer-2 firewall, digital-I/O control, and custom applications. Install only required apps and manage them as software assets. On **S1**, the only way to run scripts is to package them as **S1 Router Apps**.

## 16.7 Edge Computing

Local processing via Python, C, C++, Bash and Linux services (third-party runtimes such as Node-RED and Docker where supported). Use cases: local protocol conversion, data filtering, alarm preprocessing, local automation, cloud integration, telemetry aggregation. Local processing reduces WAN bandwidth and improves resilience.

## 16.8 Event Engine

Built-in automation that triggers on interface state, signal strength, timers, device temperature, power state, data usage and digital inputs. Actions include executing scripts, sending SNMP traps, sending email/SMS notifications, generating logs and triggering Router Apps. Automation should reduce MTTD/MTTR while remaining deterministic. (Report types include SMS, email, SNMP trap and event notifications, model/configuration dependent.)

## 16.9 Diagnostics & Firmware (in ICR-OS)

Diagnostics: System/Kernel/Reboot logs, One-Click Report, signal strength, routing table, configuration reports, remote diagnostics over SSH. Firmware: approved baseline, controlled upgrades, rollback, Change Management, version inventory, recovery validation — managed centrally via WebAccess/DMP where possible. (See *Diagnostics*, *Firmware*, *Patch Management*.)

## 16.10 Architect Notes

ICR-OS is one of the most flexible industrial router operating systems available; that openness increases architectural responsibility. Every additional service or Router App expands the attack surface. Prioritize a minimal attack surface, secure configuration, controlled software lifecycle, identity-first administration and continuous monitoring. **Operational flexibility must never compromise security** — which is precisely the trade-off the S1 build formalizes.

## 16.11 Common ICR-OS Mistakes

Treating ICR-OS as closed firmware; installing unnecessary Router Apps; unrestricted SSH; shared admin accounts; running custom apps without validation; no Change Management for Router Apps; outdated firmware; ignoring centralized monitoring.

# 17. WebAccess/DMP

## 17.1 What WebAccess/DMP Is

WebAccess/DMP (Gen2) is Advantech's **enterprise-grade Device Management Platform** for provisioning, monitoring, managing and configuring Advantech routers and IoT gateways (and Wzzard wireless sensor nodes). It is the **centralized Management Plane** for the ICR-2734. Delivered as **24/7 SaaS** (free for a small number of devices) or **on-premises (bare-metal)**, with an **HTML5 browser** front-end. It supersedes the now-EOL WebAccess/VPN and R-SeeNet.

## 17.2 Management Plane vs. Data Plane

| Management Plane (WebAccess/DMP) | Data Plane (ICR-2734) |
|---|---|
| Device inventory, identity, Configuration Profiles | Routing, NAT, firewall |
| Firmware management, VPN orchestration | VPN traffic termination |
| Monitoring, compliance, audit | Edge computing, industrial comms |

Separating these planes improves scalability and resilience. **Routers do not manage one another** — the platform controls the routers; the routers control connectivity.

## 17.3 Key Capabilities (verified)

- **Zero Touch Provisioning** / zero-touch configuration
- **Configuration Profiles** (desired-state baselines)
- **Remote firmware upgrade** and compliance reporting
- **Remote Router App management**
- **Router monitoring**, customizable/multi-dashboards, **alerts**
- **Digital Twin** model of each router (real-time visibility)
- **Geographical mapping**
- **VPN Management** — automated **WireGuard** VPN setup with a **secure proxy link** for instant **browser-based access to devices behind routers** (no manual network setup)
- **Auditing**, **multi-tenancy**, **fine-tuned permissions / RBAC**
- **PKI for mutual authorization**, **two-factor authentication**, encryption
- **API access** for integration (SIEM, CMDB, ITSM, asset management, automation)
- Ping/diagnostic tools

## 17.4 Device Lifecycle (via the platform)

```text
Procurement → Asset Registration → Zero Touch Provisioning → Configuration-Profile Assignment →
Deployment → Continuous Monitoring → Firmware Maintenance → Recovery → Retirement
```

The same lifecycle applies consistently to every managed router. Each router gains a unique identity, cryptographic trust, lifecycle record, assigned Configuration Profile and audit history.

## 17.5 Security of the Management Plane

Treat WebAccess/DMP as a **Tier-0 administrative system**: PKI, mutual authentication, TLS, RBAC, MFA, audit logging, least privilege and secure APIs. Manage API credentials under organizational identity governance.

## 17.6 Integration

APIs support integration with SIEM, CMDB, ITSM, asset management, monitoring and automation platforms. Consult the official **API reference** for automation/integration scenarios.

## 17.7 Backup & DR of the Platform

Back up Configuration Profiles, device inventory, user accounts/RBAC, audit history, VPN configuration and platform configuration; validate via Recovery Testing. For on-premises deployments, plan **Disaster Recovery of the platform itself** (availability, inventory, profiles, firmware repository, VPN orchestration, identities, audit history). (See *High Availability*.)

## 17.8 Deployment Examples

- **Water utility:** on-premises WebAccess/DMP, ~80 ICR-2734 routers, standardized Configuration Profiles, WireGuard VPN, Geo SCADA, central SIEM.
- **Manufacturing enterprise:** SaaS WebAccess/DMP, ~200 remote cells, ZTP, automated firmware deployment, centralized monitoring.
- **Critical infrastructure:** Industrial DMZ deployment, PKI authentication, RBAC, MFA, configuration-compliance monitoring, recovery-tested platform.

## 17.9 Common WebAccess/DMP Mistakes

Managing routers individually; no Configuration Profiles; manual VPN configuration; no device inventory; shared admin accounts; missing MFA; ignoring drift; no backup validation.

## 17.10 Architect Notes

The platform's greatest value is not centralized configuration — it is **centralized governance**. Every managed router becomes identifiable, measurable, auditable, recoverable and compliant, improving both cybersecurity maturity and operational efficiency.

# 18. Zero Touch Provisioning

## 18.1 What ZTP Is

Zero Touch Provisioning (ZTP) automates deployment and onboarding of ICR-2734 routers **without manual configuration at the installation site**. Using WebAccess/DMP, a newly deployed router establishes a secure connection to the platform, authenticates itself and receives its assigned configuration. ZTP is fundamentally an **identity-driven deployment architecture**, not merely an automation convenience.

## 18.2 Business Objectives

Eliminate manual configuration; standardize deployments; reduce commissioning time and configuration errors; improve security consistency; support mass deployment; simplify lifecycle management; enable rapid device replacement.

## 18.3 Provisioning Workflow

```text
Factory Reset Device → Power On → Internet Connectivity → Secure Registration →
Identity Verification → Configuration-Profile Assignment → Firmware Validation →
Router Apps Deployment → VPN Configuration → Operational Validation → Production Deployment
```

Each stage must complete successfully before the next begins; provisioning occurs **before** the router becomes part of the production OT environment.

## 18.4 Identity and Trust (the three trusted elements)

Successful ZTP depends on a **trusted device**, a **trusted management platform**, and a **trusted configuration**. Recommended trust model: unique device identity, **PKI-based mutual authentication**, TLS-protected communication, device inventory and configuration ownership. **Never** rely solely on serial numbers or shared credentials. *Automation without trust increases operational risk.*

## 18.5 What ZTP Provisions

- **Configuration Profile** (interfaces, firewall, VPN, routing, DNS, NTP, logging, monitoring, Router Apps) — managed as a controlled baseline.
- **Firmware** — validation and deployment of an approved version.
- **Router Apps** — automatic deployment under the same Change Management as firmware.
- **VPN** — automatic WireGuard/IPsec/OpenVPN setup (peers, keys, routing, firewall, DNS). VPN credentials should never be manually distributed when automated provisioning is available.

## 18.6 ZTP for Recovery / Replacement

After hardware replacement: register the replacement → verify identity → assign the previous Configuration Profile → restore firmware → restore VPN → validate connectivity → confirm operational readiness. A replacement should become operational **without manual reconfiguration** wherever possible. (See *Backup and Recovery*.)

## 18.7 Monitoring & Logging of Provisioning

Monitor provisioning status, registration failures, identity-verification failures, configuration/firmware compliance, VPN status, device health and drift. Log initial registration, configuration deployment, firmware installation, configuration changes, provisioning failures, identity verification and administrator actions. Forward provisioning events to centralized monitoring/SIEM.

## 18.8 Common ZTP Mistakes

Manual router configuration; shared configuration files; shared admin accounts; missing firmware validation; missing configuration baselines; ignoring provisioning failures; no inventory synchronization; no audit trail.

## 18.9 Decision Support (ZTP)

Which identity mechanism authenticates devices? Which Configuration Profiles are required? Which firmware versions are approved? Which Router Apps deploy automatically? Which VPN topology is provisioned? How is compliance monitored? How are replacement devices onboarded and audited?
